Q2. What is the difference between a slice and an array in Go?
------------------------------------------------------------------------------------------
TOP SCORE : 0.89
SOURCES   : Go Slices: usage and internals (0.89), Arrays, slices (and
            strings): The mechanics of 'append' (0.81), (same) (0.81)
ANSWER    : The main difference between a slice and an array in Go is that an array has a 
            specified length, which is part of its type, whereas a slice does not have a 
            specified length [1][3]. This means that arrays are less flexible and have limited 
            expressive power compared to slices [1][3]. Additionally, when slice values are 
            passed around, the header gets copied but the underlying array is shared, 
            whereas arrays are not shared in the same way [2].

ANSWER    : Gofmt is a tool that automatically formats Go source code [1][3]. 
            It makes code easier to write, read, and maintain by standardizing the formatting 
            style, allowing developers to focus on the code itself rather than formatting concerns [1]. 
            This also helps with version control, as mechanical changes to the source code 
            won't cause unrelated changes to the file's formatting, making diffs more accurate [1]. 
            Additionally, using gofmt helps keep code in the canonical style, 
            which can be easily enforced with hooks for editors and version control systems [3].

 SHORT DISCUSSION
==========================================================================================

WHAT WORKED
  - Generation is reliably grounded. Across all 8 queries the model never
    invented facts. When the retrieved chunks fully covered the question
    (Q1-Q4) the answers were accurate and cited the correct sources; when
    they did not (Q5-Q7) the model said so plainly instead of guessing.
    This is exactly the behaviour that separates a RAG system from a
    general chatbot.
  - Graceful failure works. The out-of-domain query (Q8) was declined by
    the relevance gate. In-domain queries score 0.54-0.89 and out-of-
    domain queries score ~0.20, so the 0.35 threshold cleanly separates
    them (see the chunk-size sweep in the README for how it was chosen).
  - Citations map to the sources panel. The [n] numbers in each answer
    correspond to the numbered chunks shown in the UI, so every claim is
    traceable back to a passage the user can expand and read.

WHERE IT STRUGGLED
  - Retrieval recall with small top_k (Q5). A question whose answer is
    spread across several passages (the three laws of reflection) is only
    partially answered at top_k=3, because not every relevant chunk fits
    in the window. Raising top_k to ~5 fixes it at a small latency cost.
  - Semantic drift in ranking (Q6). The embedding model occasionally
    ranks a topically-adjacent chunk above the truly relevant one
    ("Laws of Reflection" outranking the generics article because both
    discuss "types"). A cross-encoder re-ranking pass over the top
    candidates would correct this.
  - Chunk granularity (Q7). Retrieval can land on the right *document* but
    the wrong *chunk*, so a specific how-to (JSON Unmarshal into a struct)
    is missed even though the article covers it. Section-aware chunking or
    a larger top_k would improve coverage.

TAKEAWAY
  Generation is the strong, trustworthy layer; retrieval is the current
  bottleneck. The three partial results (Q5-Q7) are all retrieval-recall
  or ranking issues, not hallucinations — which is the safe way for a RAG
  system to fail. The highest-value next improvements are all on the
  retrieval side: a slightly larger top_k, a cross-encoder re-ranker, and
  section-aware chunking.
